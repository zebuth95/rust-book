# Fundamentals of Asynchronous Programming: Async, Await, Futures, and Streams

Rust's async model — **cooperative multitasking** for high-performance I/O. Multiple tasks run on a single thread, switching at `.await` points.

| Concept | Purpose |
|---------|---------|
| **`Future` trait** | An async computation that may not be ready yet (`Poll::Pending` or `Poll::Ready`). |
| **`async` / `.await`** | Sugar for writing and composing futures. |
| **`Stream` trait** | An async iterator — values arrive over time. |
| **Executor (e.g. Tokio)** | Polls futures, wakens them when ready, switches tasks at await points. |

---

## Quick Example

```rust
use tokio::time::{sleep, Duration};

async fn fetch_data(id: u32) -> String {
    sleep(Duration::from_millis(100)).await;
    format!("Data from task {id}")
}

#[tokio::main]
async fn main() {
    let (a, b) = tokio::join!(fetch_data(1), fetch_data(2));
    println!("{a}, {b}"); // Both finish in ~100ms, not 200ms
}
```

## Key Points

- **Not parallelism** — async is about *interleaving* tasks, not running them simultaneously on multiple cores.
- **Zero-cost** — async state machines are compiled at compile time; no allocation per task.
- **Best for I/O** — networking, file reads, database queries. Not for CPU-heavy work (use threads for that).
- **Needs a runtime** — `async fn` alone does nothing; an executor (Tokio, async-std, smol) must poll the future.
- **Cancellation safety** — When a future is dropped (e.g., by `select!`), in-flight operations may be lost. Use `CancellationToken` for graceful cancellation.
- **`tokio::spawn`** — Creates a new task on the runtime. The future must be `'static + Send`.
