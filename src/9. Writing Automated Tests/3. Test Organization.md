# Test Organization

Rust distinguishes between two main types of tests based on file placement, leveraging the module system.

## Unit Tests:
Test individual units of code in isolation (e.g., a function or method). They are typically placed in a mod tests submodule within the same file as the code they are testing. This submodule is marked with #[cfg(test)] so it's only compiled when running tests. Unit tests can test private functions.

## Integration Tests:
Test how multiple parts of the public API interact with each other. They live in the tests directory next to src. Each file in this directory is treated as a separate crate, and they can only test public functions.

### Unit Test Location (in src/lib.rs):

    pub fn add_two(a: i32) -> i32 { a + 2 }
    
    #[cfg(test)]
    mod tests {
    use super::*; // Bring parent items (like add_two) into scope
    
        #[test]
        fn it_adds_two() {
            assert_eq!(4, add_two(2));
        }
    }

### Integration Test Location (in tests/integration_test.rs):

    // This file is treated as an external crate
    use my_project; // Assuming the package is named 'my_project'
    
    #[test]
    fn it_adds_two_via_lib() {
        // Only public functions from the library can be tested here
        assert_eq!(4, my_project::add_two(2));
    }
