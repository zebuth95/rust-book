# Error Handling

Two categories of errors:

| Type | What it is | How to handle |
|---|---|---|
| **Recoverable** | File not found, network timeout | `Result<T, E>` |
| **Unrecoverable** | Array out of bounds, bug | `panic!` |

Rust has **no exceptions**. Use `Result` for expected failures, `panic!` for bugs.

## Error Crate Ecosystem

| Crate | Use Case |
|-------|----------|
| **`thiserror`** | Derive `Error` for library error types (custom enums) |
| **`anyhow`** | Ad-hoc error handling in applications (Box<dyn Error> sugar) |
| **`eyre`** | Flexible error reporting with customizable reports (like anyhow but more configurable) |

### `thiserror` Example (Libraries)

```rust
use thiserror::Error;

#[derive(Error, Debug)]
pub enum DataError {
    #[error("data store disconnected")]
    Disconnect(#[from] io::Error),
    #[error("invalid header: {0:?}")]
    InvalidHeader(Vec<u8>),
    #[error("unknown error")]
    Unknown,
}
```

### `anyhow` Example (Applications)

```rust
use anyhow::{Result, Context};

fn read_config() -> Result<String> {
    let content = std::fs::read_to_string("config.toml")
        .with_context(|| "failed to read config file")?;
    Ok(content)
}
```

> **Note:** Use `thiserror` in libraries (callers decide how to handle). Use `anyhow` in applications (just report and move on).
