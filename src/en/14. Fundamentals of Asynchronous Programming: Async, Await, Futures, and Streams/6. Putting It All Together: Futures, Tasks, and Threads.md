# Putting It All Together: Futures, Tasks, and Threads

The async runtime stacks three layers to deliver efficient concurrency.

## The Three Layers

```
OS Thread (heavy, limited)
    └── Executor (Tokio runtime)
            └── Many Tasks (lightweight, scheduled by runtime)
                    └── Each Task wraps one Future (the state machine)
```

| Layer | Description | Cost |
|-------|-------------|------|
| **OS Thread** | Real operating system thread. | High (1+ MB stack, kernel scheduling) |
| **Task** | A future managed by the runtime. Lightweight, can sleep/yield. | Very low (allocation + state) |
| **Future** | State machine generated by `async fn`. | Zero (stack-only, no allocation) |

## How Execution Works

```rust
#[tokio::main]
async fn main() {
    // Task 1: spawned onto the executor
    let handle = tokio::spawn(async {
        tokio::time::sleep(std::time::Duration::from_millis(100)).await;
        "done"
    });

    // Task 2: runs concurrently on the same thread
    let result = tokio::time::timeout(
        std::time::Duration::from_secs(1),
        handle,
    )
    .await;

    println!("{:?}", result);
}
```

## The Execution Flow

```
1. main() starts → creates a Task wrapping main's future
2. tokio::spawn() creates a second Task
3. Thread polls Task 1 (main) → hits .timeout().await → Pending
4. Thread switches to Task 2 (spawned) → hits .sleep().await → Pending
5. 100ms later → sleep wakes Task 2 → Thread polls it → Ready("done")
6. Task 2 Ready wakes Task 1 → Thread polls main → Ready
```

## When to Use Async vs Threads

| Scenario | Use |
|----------|-----|
| **I/O-bound** (network, files, DB) | Async — many operations waiting concurrently |
| **CPU-bound** (computation, rendering) | Threads (`std::thread`) — need real parallelism |
| **Mixed** | Async + `spawn_blocking` or `rayon` for CPU work |

---

## Key Points

| Concept | Meaning |
|---------|---------|
| **Task** | A spawned `Future` managed by the runtime. Runs on a thread pool. |
| **Executor** | The runtime (Tokio) that polls tasks and handles wakeups. |
| **Cooperative scheduling** | Tasks yield at `.await` — no preemption. |
| **`tokio::spawn`** | Creates a new task on the same executor. |
| **Many:1 mapping** | Thousands of tasks can run on a handful of threads. |

#### Cancellation Safety

When a future is dropped (e.g., `select!` wins another branch), any in-progress operations inside it are **cancelled**. This can cause data loss.

**Pattern: Graceful Shutdown with CancellationToken**

```rust
use tokio_util::sync::CancellationToken;

async fn worker(token: CancellationToken) {
    tokio::select! {
        _ = token.cancelled() => {
            println!("Worker shutting down gracefully");
            // flush state, close connections, etc.
        }
        _ = async {
            // normal work
            loop { tokio::time::sleep(Duration::from_secs(1)).await; }
        } => {}
    }
}

#[tokio::main]
async fn main() {
    let token = CancellationToken::new();
    let handle = tokio::spawn(worker(token.clone()));
    token.cancel(); // triggers graceful shutdown
    let _ = handle.await;
}
```

#### `tracing` Instrumentation

Add observability to async code:

```rust
#[tracing::instrument]
async fn handle_request(req: Request) -> Response {
    tracing::info!("processing request");
    // ...
}
```

- `tracing` integrates with OpenTelemetry for distributed tracing.
- Use `tokio-console` to visualise task states in development.

#### Avoiding `Mutex` Across `.await`

```rust
// ❌ BAD: Holding lock across .await — blocks the runtime
async fn bad() {
    let data = Arc::new(Mutex::new(vec![1, 2, 3]));
    let mut guard = data.lock().unwrap();
    some_async_fn().await;  // lock held! other tasks can't access data
    guard.push(4);
}

// ✅ GOOD: Scoped lock — released before .await
async fn good() {
    let data = Arc::new(Mutex::new(vec![1, 2, 3]));
    {
        let mut guard = data.lock().unwrap();
        guard.push(4);
    } // lock released here
    some_async_fn().await; // no lock held
}

// ✅ EVEN BETTER: Use tokio::sync::Mutex for longer holds
use tokio::sync::Mutex;
async fn better() {
    let data = Arc::new(Mutex::new(vec![1, 2, 3]));
    let mut guard = data.lock().await;
    some_async_fn().await;  // ✅ Tokio Mutex is safe across .await
    guard.push(4);
}
```
