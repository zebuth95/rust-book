# Refutability: Whether a Pattern Might Fail to Match
**Refutability** determines whether a petter can fail to match a value.

- Irrefutable Pattern: A pattern that always matches the value, no matter the value. (e.g., x, (x, y), Some(x) if the value is guaranteed to be an Option<T>).
- Refutable Pattern: A pattern that might fail to match the value. (e.g., Some(x), EnumVariant::B).

## Rule:
Constructs that require exhaustive matching (like let statements and function parameters) only accept irrefutable patterns. Constructs designed for conditional matching (like if let and match arms) accept refutable patterns.

# Pattern Type Allowances

| Context            | Accepts            | Reason                    |
|--------------------|--------------------|---------------------------|
| let statements     | Irrefutable only   | Must always succeed       |
| Function parameters | Irrefutable only   | Must always succeed       |
| if let            | Refutable          | Conditional matching      |
| while let         | Refutable          | Conditional looping       |
| match arms        | Both               | Designed for pattern matching |


### Irrefutable Patterns (Always Work)

    // These ALWAYS match and cannot fail
    let x = 5;                    // ✅ Always matches
    let (a, b) = (1, 2);          // ✅ Always matches tuples
    let [first, second] = [1, 2]; // ✅ Always matches arrays of correct length
    
    fn print_coordinates(&(x, y): &(i32, i32)) {  // ✅ Function parameter
        println!("Location: ({}, {})", x, y);
    }

### Refutable Patterns (Might Fail)

    let some_option: Option<i32> = Some(5);
    let none_option: Option<i32> = None;
    
    // ❌ COMPILER ERROR: 'let' requires irrefutable pattern
    // let Some(x) = some_option;  // This would fail if value was None!
    
    // ✅ Correct: Using conditional constructs for refutable patterns
    if let Some(x) = some_option {
    println!("Got value: {}", x);  // Prints: Got value: 5
    }
    
    if let Some(x) = none_option {
        println!("This won't print");  // Doesn't execute
    } else {
        println!("It was None!");      // Prints: It was None!
    }
    
    // Using while let with refutable pattern
    let mut values = vec![Some(1), Some(2), None, Some(3)].into_iter();
    while let Some(Some(value)) = values.next() {
        println!("Processing: {}", value);  // Prints: Processing: 1, Processing: 2
    }

### Match Expressions (Both Types)

    fn process_value(value: Option<i32>) {
        match value {
            // Refutable patterns in match arms
            Some(42) => println!("The answer!"),
            Some(x) if x > 0 => println!("Positive: {}", x),
            Some(0) => println!("Zero"),
            Some(_) => println!("Negative"),
            
            // This arm uses an irrefutable pattern
            x => println!("Something else: {:?}", x),  // Always matches
        }
    }
    
    process_value(Some(42));  // "The answer!"
    process_value(Some(10));  // "Positive: 10"  
    process_value(None);      // "Something else: None"

### Common Patterns and Their Refutability

    // IRREFUTABLE patterns:
    let x = value;           // ✅ Variable binding
    let (a, b, c) = tuple;   // ✅ Tuple destructuring
    let Point { x, y } = point;  // ✅ Struct destructuring (all fields)
    
    // REFUTABLE patterns:
    if let Some(x) = opt { }     // ✅ Option matching
    if let Ok(result) = res { }  // ✅ Result matching  
    if let 1..=5 = num { }      // ✅ Range matching
    if let 'a'..='z' = ch { }   // ✅ Char range matching

### Practical Example: Handling Different Scenarios

    enum Message {
        Quit,
        Move { x: i32, y: i32 },
        Write(String),
        ChangeColor(i32, i32, i32),
    }
    
    fn handle_message(msg: Message) {
        // ❌ Would cause error - refutable pattern in let
        // let Message::Move { x, y } = msg;
    
        // ✅ Correct - using match for refutable patterns
        match msg {
            Message::Quit => println!("Quit message"),
            Message::Move { x, y } => println!("Move to ({}, {})", x, y),
            Message::Write(text) => println!("Text message: {}", text),
            Message::ChangeColor(r, g, b) => println!("Change color to RGB({}, {}, {})", r, g, b),
        }
        
        // ✅ Using if let for specific cases
        if let Message::Write(text) = msg {
            println!("Specifically handling Write: {}", text);
        }
    }
    
    // Function parameters use irrefutable patterns
    fn process_move(Message::Move { x, y }: Message::Move) {  // ✅
        println!("Processing move to ({}, {})", x, y);
    }

### Key Takeaways

- let statements need certainty - they must always succeed, so they only accept irrefutable patterns
- Conditional constructs handle uncertainty - if let, while let, and match are designed for patterns that might fail
- Compiler protects you - Rust won't compile code that tries to use refutable patterns where irrefutable ones are required
- Choose the right tool - Use let for guaranteed matches, conditional constructs for possible failures

This system prevents runtime errors by ensuring patterns that might fail are only used in contexts that can handle the failure!
