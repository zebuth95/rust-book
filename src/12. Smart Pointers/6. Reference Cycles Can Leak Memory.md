# Reference Cycles Can Leak Memory
Rut's memory safety guarantees make it difficult, but not impossible, to accidentally create memory that is never clean up
(known as a memory leak). We can see that Rust allows memory leaks by using **Rc<T>** and **RefCell<T>**:
it's possible ro create references where items refer ot each other in a cycle.
This creates memory leaks because the reference count of each item in the cycle neve reach 0, and the values will never be dropped.

#### When using Rc<T>, it is possible to create a reference cycle: two or more Rc<T> instances that directly or indirectly refer to each other.

### Example (Conceptual Cycle):
A node A holds an Rc reference to node B, and node B holds an Rc reference back to node A. Both counts remain at 1, preventing cleanup.

## Problem:
If a cycle exists, the reference count for the data will never reach zero, even if the user is no longer using the data. This results in a memory leak, as the Drop trait implementation is never called.

## Solution:
Use Weak<T> instead of Rc<T> for back references or parent pointers. Weak<T> references do not count toward the reference count, allowing the cycle to be broken and the memory to be properly cleaned up.

## Fix Example (Using Weak):

    use std::rc::{Rc, Weak};
    use std::cell::RefCell;
    
    struct Node {
        value: i32,
        parent: RefCell<Weak<Node>>, // Weak reference for parent
        children: RefCell<Vec<Rc<Node>>>,
    }
    // The Weak reference allows the parent to be cleaned up independently
