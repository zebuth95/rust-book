# Poniendo Todo Junto: Futures, Tareas y Hilos

El runtime asíncrono apila tres capas para proporcionar concurrencia eficiente.

## Las Tres Capas

```
OS Thread (heavy, limited)
    └── Executor (Tokio runtime)
            └── Many Tasks (lightweight, scheduled by runtime)
                    └── Each Task wraps one Future (the state machine)
```

| Capa | Descripción | Costo |
|-------|-------------|------|
| **Hilo del SO** | Hilo real del sistema operativo. | Alto (1+ MB de pila, planificación del kernel) |
| **Tarea** | Un future gestionado por el runtime. Ligero, puede dormir/ceder. | Muy bajo (asignación + estado) |
| **Future** | Máquina de estado generada por `async fn`. | Cero (solo pila, sin asignación) |

## Cómo Funciona la Ejecución

```rust
#[tokio::main]
async fn main() {
    // Task 1: spawned onto the executor
    let handle = tokio::spawn(async {
        tokio::time::sleep(std::time::Duration::from_millis(100)).await;
        "done"
    });

    // Task 2: runs concurrently on the same thread
    let result = tokio::time::timeout(
        std::time::Duration::from_secs(1),
        handle,
    )
    .await;

    println!("{:?}", result);
}
```

## El Flujo de Ejecución

```
1. main() starts → creates a Task wrapping main's future
2. tokio::spawn() creates a second Task
3. Thread polls Task 1 (main) → hits .timeout().await → Pending
4. Thread switches to Task 2 (spawned) → hits .sleep().await → Pending
5. 100ms later → sleep wakes Task 2 → Thread polls it → Ready("done")
6. Task 2 Ready wakes Task 1 → Thread polls main → Ready
```

## Cuándo Usar Async vs Hilos

| Escenario | Usar |
|----------|-----|
| **I/O-bound** (red, archivos, BD) | Async — muchas operaciones esperando concurrentemente |
| **CPU-bound** (cómputo, renderizado) | Hilos (`std::thread`) — se necesita paralelismo real |
| **Mixto** | Async + `spawn_blocking` o `rayon` para trabajo de CPU |

---

## Puntos Clave

| Concepto | Significado |
|---------|---------|
| **Tarea** | Un `Future` lanzado gestionado por el runtime. Se ejecuta en un pool de hilos. |
| **Executor** | El runtime (Tokio) que hace poll a las tareas y maneja los despertares. |
| **Planificación cooperativa** | Las tareas ceden en `.await` — sin preferencia. |
| **`tokio::spawn`** | Crea una nueva tarea en el mismo executor. |
| **Mapeo muchos:1** | Miles de tareas pueden ejecutarse en un puñado de hilos. |

#### Seguridad de Cancelación

Cuando un future se descarta (ej., `select!` gana otra rama), cualquier operación en curso dentro de él se **cancela**. Esto puede causar pérdida de datos.

**Patrón: Apagado Gradual con CancellationToken**

```rust
use tokio_util::sync::CancellationToken;

async fn worker(token: CancellationToken) {
    tokio::select! {
        _ = token.cancelled() => {
            println!("Worker shutting down gracefully");
            // flush state, close connections, etc.
        }
        _ = async {
            // normal work
            loop { tokio::time::sleep(Duration::from_secs(1)).await; }
        } => {}
    }
}

#[tokio::main]
async fn main() {
    let token = CancellationToken::new();
    let handle = tokio::spawn(worker(token.clone()));
    token.cancel(); // triggers graceful shutdown
    let _ = handle.await;
}
```

#### Instrumentación con `tracing`

Añade observabilidad al código asíncrono:

```rust
#[tracing::instrument]
async fn handle_request(req: Request) -> Response {
    tracing::info!("processing request");
    // ...
}
```

- `tracing` se integra con OpenTelemetry para trazado distribuido.
- Usa `tokio-console` para visualizar estados de tareas en desarrollo.

#### Evitando `Mutex` a través de `.await`

```rust
// ❌ BAD: Holding lock across .await — blocks the runtime
async fn bad() {
    let data = Arc::new(Mutex::new(vec![1, 2, 3]));
    let mut guard = data.lock().unwrap();
    some_async_fn().await;  // lock held! other tasks can't access data
    guard.push(4);
}

// ✅ GOOD: Scoped lock — released before .await
async fn good() {
    let data = Arc::new(Mutex::new(vec![1, 2, 3]));
    {
        let mut guard = data.lock().unwrap();
        guard.push(4);
    } // lock released here
    some_async_fn().await; // no lock held
}

// ✅ EVEN BETTER: Use tokio::sync::Mutex for longer holds
use tokio::sync::Mutex;
async fn better() {
    let data = Arc::new(Mutex::new(vec![1, 2, 3]));
    let mut guard = data.lock().await;
    some_async_fn().await;  // ✅ Tokio Mutex is safe across .await
    guard.push(4);
}
```
