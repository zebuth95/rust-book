# Fundamentos de Programación Asíncrona: Async, Await, Futures y Streams

El modelo asíncrono de Rust — **multitarea cooperativa** para I/O de alto rendimiento. Múltiples tareas se ejecutan en un solo hilo, cambiando en los puntos `.await`.

| Concepto | Propósito |
|---------|---------|
| **`Future` trait** | Una computación asíncrona que puede no estar lista aún (`Poll::Pending` o `Poll::Ready`). |
| **`async` / `.await`** | Azúcar sintáctico para escribir y componer futures. |
| **`Stream` trait** | Un iterador asíncrono — los valores llegan a lo largo del tiempo. |
| **Executor (ej. Tokio)** | Hace poll a los futures, los despierta cuando están listos, cambia de tarea en los puntos await. |

---

## Ejemplo Rápido

```rust
use tokio::time::{sleep, Duration};

async fn fetch_data(id: u32) -> String {
    sleep(Duration::from_millis(100)).await;
    format!("Data from task {id}")
}

#[tokio::main]
async fn main() {
    let (a, b) = tokio::join!(fetch_data(1), fetch_data(2));
    println!("{a}, {b}"); // Both finish in ~100ms, not 200ms
}
```

## Puntos Clave

- **No es paralelismo** — async se trata de *entrelazar* tareas, no de ejecutarlas simultáneamente en múltiples núcleos.
- **Zero-cost** — las máquinas de estado async se compilan en tiempo de compilación; sin asignación por tarea.
- **Mejor para I/O** — redes, lectura de archivos, consultas a bases de datos. No para trabajo intensivo de CPU (usa hilos para eso).
- **Necesita un runtime** — `async fn` solo no hace nada; un executor (Tokio, async-std, smol) debe hacer poll al future.
- **Seguridad de cancelación** — Cuando un future se descarta (ej. por `select!`), las operaciones en curso pueden perderse. Usa `CancellationToken` para cancelación gradual.
- **`tokio::spawn`** — Crea una nueva tarea en el runtime. El future debe ser `'static + Send`.
